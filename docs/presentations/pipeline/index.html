ExamShield is an enterprise-grade, AI-powered secure exam scanning system. This presentation walks through the full 11-stage pipeline — from the moment an invigilator photographs an answer sheet to the moment results are published.

The Problem

Paper Exam Risks Today

  • Answer sheets photographed or replaced after the exam
  • OCR results edited before scoring
  • No cryptographic proof of when a photo was taken
  • No audit trail linking device → invigilator → sheet → score
  • Human reviewer can silently "correct" OCR answers
  • Results can be modified before publication

What's Missing

  • On-device cryptographic commitment of the image
  • Server-side independent verification
  • Immutable append-only evidence chain
  • Role separation — capture ≠ review ≠ score ≠ publish
  • Encryption at rest — storage access yields only ciphertext
  • Anonymous public verification of any result
Traditional paper exam workflows have no digital chain of custody. ExamShield solves this with cryptographic commitments at every stage.

Stages 2–3 of 11

Hash & Sign

Role: Device Crypto (on-device)

Hashing

  • SHA-256 of raw JPEG bytes — computed before any network call
  • Deterministic: same bytes → same hash every time
  • Any post-capture modification changes the hash
final hash = sha256.convert(imageBytes);
// e3b0c44298fc1c149afb...

Signing

  • ECDSA P-256 or Ed25519 with device private key
  • Private key stored in device secure enclave — never transmitted
  • Signature binds device identity to the hash
final sig = signer.sign(hash.bytes);
// Device key pair registered at enrollment
Security invariant: the server cannot trust a hash the client claims — it re-computes independently. If they differ, the upload is permanently rejected.
The hash-then-sign pattern is the core cryptographic anchor of the whole pipeline. SHA-256 commits the image content; ECDSA binds the device identity to that commitment.
☁️✅
Stages 4–5 of 11

Upload & Verify

Role: Invigilator → Server

Two-Phase Upload

  • POST /capture — register hash, signature, metadata
  • POST /upload — send image bytes
  • Phase 1 creates an auditable record before any bytes arrive
  • Retries are idempotent — same capture ID, no duplicate records

Server Verification (critical gate)

  • Re-compute SHA-256 of received bytes server-side
  • Compare against registered hash — any mismatch → reject
  • Verify ECDSA signature against registered public key
  • Unknown device → 403 + SECURITY_ALERT

The two-phase upload prevents orphaned uploads and makes every transfer auditable. The verification gate is the most critical security check in the pipeline.

Stages 6–7 of 11

Watermark & Encrypt

Role: Server Pipeline

Invisible Watermark

  • Steganographic — imperceptible to the human eye
  • Encodes: examId, timestamp, nonce, original hash, deviceId
  • Forensic tools extract + validate the embedded payload
  • Watermark destruction = tampering signal

AES-256-GCM Encryption at Rest

  • Fresh 256-bit DEK generated per image at upload
  • DEK is envelope-encrypted with master key (KMS/Vault in prod)
  • Ciphertext + encrypted DEK stored in MinIO / S3
  • Direct object storage access → ciphertext only
Key hierarchy: ImageBytes → AES-256-GCM(DEK) → Ciphertext ‖ DEK → AES-256-GCM(MasterKey) → EncryptedDEK (stored in DB)
Two independent post-verification protections: watermarking catches downstream pixel-level tampering, encryption makes raw storage access useless without the master key.

Stage 8 of 11

OCR Pipeline

Role: OCR Engine (system)
Data In
Decrypted image bytes (transient)
→
Data Out
Answers + confidence scores per question

Processing Steps

  • Decrypt image transiently (never stored unencrypted)
  • Detect bubble grid positions (OpenCV)
  • Extract fill-level per bubble (ML Kit / Tesseract)
  • Assign confidence score (0.0 – 1.0) per question

Routing Logic

  • High confidence (≥ threshold) → auto-accepted
  • Low confidence → Manual Review queue
  • No answer is silently discarded or guessed
  • OCR Engine cannot modify the original stored image
SoD: The OCR Engine is a system role — it cannot publish results, influence scoring, or create exam records. Its only capability is reading images and writing OCR result records.
OCR Engine is a system principal (non-human JWT) with narrowly scoped permissions. The confidence routing ensures no answer is ever silently discarded.
👁
Stage 9 of 11

Manual Review

Role: Manual Reviewer + Review Supervisor

What Reviewers Can Do

  • View the original, unmodified answer sheet image
  • See OCR predictions and confidence scores side-by-side
  • Select their interpretation of the student's answer
  • Escalate disputed items to Review Supervisor

Pixel Lock — Hard Constraints

  • Cannot paint, erase, or modify any pixel
  • Cannot delete or replace the original image
  • Cannot see another reviewer's in-progress decisions
  • Every decision creates an immutable audit record
Immutability guarantee: the reviewer's action only appends a new interpretation record — the original image and OCR result are never modified. Reviewers cannot influence the original evidence.
The Manual Review UI enforces pixel lock at the application level. The backend additionally rejects any write to Capture or OCRResult records from Reviewer-role JWTs.
⭐📢
Stages 10–11 of 11

Score & Publish

Scoring Engine

  • Reads only finalized, confirmed answer interpretations
  • Compares against official answer key
  • Produces score, percentile, and statistics
  • Cannot alter answer interpretations or images

Result Publisher

  • Triggers or schedules result release
  • Generates official downloadable reports
  • Cannot modify scores — read-only to scoring output
  • Publication event logged with publisher identity

Tamper Detection

What happens at every attack surface

Attack: Modified image in transit

Original: SHA-256 = a7f3…
↓ attacker modifies 1 pixel
Server re-hash = 9c2b… ≠ a7f3
❌ VERIFICATION_FAILED → 400 + alert

Attack: Replaced stored image

Stored watermark: examId + hash
↓ attacker replaces file in MinIO
Forensic watermark extraction: mismatch
❌ WATERMARK_TAMPER_DETECTED → alert
Attack: Unregistered device
Signature verification fails → 403 + DEVICE_NOT_TRUSTED security event
Attack: Reviewer edits a pixel
API rejects write to Capture; UI enforces pixel lock; any attempt logged as TAMPERING_ATTEMPT
The pipeline has multiple independent detection layers — SHA-256 hash check, signature verification, watermark validation, and pixel lock enforcement. Every detection triggers a security alert to the Security Center.

8 Core Security Invariants

These are never compromised — they are enforced at the domain and infrastructure layers

1 Immutability — stored images and audit logs are append-only; no update or delete path exists
2 Hash-then-Sign — SHA-256 computed on-device before any network call; server re-verifies on receipt
3 Digital Signatures — every capture signed with device private key; server rejects unknown or mismatched keys
4 Invisible Watermark — embedded in every stored image; watermark destruction signals tampering
5 Read-Only Review — manual reviewers record interpretation only; original image is unmodifiable
6 Chain of Custody — every state transition signed by the responsible actor and logged
7 Encryption at Rest — AES-256-GCM per-image DEK; raw storage access yields only ciphertext
8 Image Access Allowlist — only 6 specific roles may retrieve image bytes; admins and auditors receive 403
These invariants are the non-negotiable contracts of the system. They are enforced at both the domain layer (C# domain model) and the infrastructure layer (EF Core, MinIO storage, authorization policies).
